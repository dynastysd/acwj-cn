# 第 22 章：局部变量和函数调用的设计思路

这是编译器编写旅程中第一个不引入新代码的部分。这一次，我需要从程序员的键盘前退后一步，从宏观的角度来审视整个项目。这让我有机会思考如何实现局部变量（在某一部分），以及函数实参和形参（在下一部分）。

这两个步骤都会对我们的现有编译器进行一些重要的补充和修改。我们还需要处理一些新概念，比如栈帧和寄存器溢出，到目前为止我一直没有涉及这些内容。

让我们首先确定我们想要为编译器添加哪些新功能。

## 我们需要什么功能

### 局部和全局变量作用域

目前，所有的变量都对所有函数全局可见。我们希望为变量添加局部作用域，让每个函数拥有自己的变量，这些变量不能被其他函数看到。此外，对于递归函数，同一个函数的每个实例都有自己的局部变量。

但是，我只打算添加两种作用域：局部和全局。C 语言实际上为每个复合语句创建一个新作用域。在下面的例子中，有三个不同作用域中的三个不同的 `a` 变量：

```c
#include <stdio.h>
int a = 2;              // 全局作用域

int main()
{
  int a= 5;             // 局部作用域
  if (a > 2) {
    int a= 17;          // 第三层作用域
    printf("%d\n", a);  // 打印 17
  }
  printf("%d\n", a);    // 打印 5
  return(0);
}
```

我不打算支持第三层、最内层的作用域。两种就够了！

### 作为局部变量的函数形参

我们还需要支持声明零个或多个函数形参，这些形参需要被视为该函数实例的局部变量。

C 函数是传值调用：函数调用者的实参值被复制到函数的形参中，这样被调用的函数就可以使用和修改它们。

### 引入栈

为了给同一个函数的多个实例创建局部作用域，并为函数的形参提供存储位置，我们需要一个栈。如果你对栈不太了解，应该先做一些背景阅读。我会从这个关于调用栈的维基百科文章开始。

鉴于我们支持的硬件架构之一是运行 Linux 的 Intel x86-64 架构，我们将不得不在这个架构上实现函数调用机制。我发现 Eli Bendersky 有一篇很棒的文章关于 x86-64 上的栈帧布局。在继续阅读本文档之前，你绝对需要阅读这篇文章！由于 Eli's 的文章处于公共领域，我在此复制他的栈帧和寄存器中参数的图片：

```c
long myfunc(long a, long b, long c, long d,
            long e, long f, long g, long h)
{
    long xx, yy, zz;
    ...
}
```

![](Figs/x64_frame_nonleaf.png)

基本上，在 x86-64 架构上，一些形参的值会通过寄存器传递，一些形参的值会被压入栈中。所有的局部变量都会在栈上，但位于栈基指针下方。

同时，我们希望我们的编译器能够移植到不同的架构。因此，我们需要为不同的架构支持一个通用的函数形参框架，这些架构可能只使用栈、只使用寄存器或两者的组合。

### 寄存器溢出

到目前为止我一直忽略且尚未实现的事情是寄存器溢出。我们需要溢出我们已分配的一些或全部寄存器，原因如下：

  + 我们已用完了可分配的寄存器，因为寄存器的数量是固定的。我们可以将一个寄存器溢出到栈上，这样它就空闲可分配了。
  + 在函数调用之前，我们需要将所有已分配的寄存器以及带有形参的寄存器都溢出到栈上。这释放了它们，这样被调用的函数就可以使用它们了。

函数调用返回时，我们需要取消溢出寄存器以获取所需的值。同样，如果我们溢出了一个寄存器以使其空闲，那么当它再次空闲时，我们需要取消溢出其旧值并重新分配它。

### 静态变量

虽然不是立即要实现的事项列表，但某个时候我需要分配静态变量。局部静态变量会有一些命名问题，但在实现所有这些即时的想法时，我会尽量记住这一点。

### 变量初始化

我们应该允许在声明变量时对其进行初始化。对于全局变量，我们肯定可以将它们初始化为常量值，例如 `int x= 7;`，但不能初始化为表达式，因为我们没有可以运行初始化代码的函数上下文。

但是，我们应该能够进行局部变量初始化，例如 `int a= 2, b= a+5;`，因为我们可以将变量的初始化代码插入到函数代码的开头。

## 想法与实现

好的，这些是我目前脑海中正在酝酿的设计想法和问题的。以下是我认为如何实现其中一些的方法。

### 局部符号

让我们从区分局部变量和全局变量开始。全局变量必须对所有函数可见，但局部变量只对一个函数可见。

SubC 使用一个符号表来存储有关局部和全局变量的信息。全局变量分配在一端，局部变量存储在另一端。有代码确保两端在中间不会发生冲突。我喜欢这个想法，因为这样我们就可以为每个符号拥有一组唯一的符号槽号，无论其作用域如何。

在优先处理局部符号而不是全局符号方面，我们可以先搜索符号表的局部端，如果我们找不到符号，然后再搜索全局端。而且，一旦我们完成解析一个函数，就可以简单地清除符号表的局部端。

### 存储类

C 有存储类的概念，我们必须至少实现其中一些。SubC 实现了多个存储类：

```c
/* storage classes */
enum {
        CPUBLIC = 1,            // 公开可见的符号
        CEXTERN,                // extern 符号
        CSTATIC,                // 全局上下文中的静态符号
        CLSTATC,                // 局部上下文中的静态符号
        CAUTO,                  // 非静态的局部标识符
        CSPROTO,                // 函数原型
        CMEMBER,                // 结构体/联合体的字段
        CSTCDEF                 // 未使用
};
```

对于符号表中的每个符号。我认为我可以修改并使用这个，但我可能会支持更少的存储类类型。

### 函数原型

每个函数都有一个原型：其每个形参的数量和类型。我们需要这些来确保函数调用的实参与形参的类型和数量相匹配。

我需要在某处记录每个函数的形参列表和类型。我们还可以支持在函数实际声明之前声明函数的原型。

现在，我们要把它存储在哪里？我可以为函数原型创建一个单独的数据结构。我不想在我们的语言中支持二维数组，但我们需要每个函数一个原始类型的列表。

所以，我的想法是这样的。我们已经有了 S_FUNCTION 作为现有符号表元素的类型。我们可以在每个符号表条目中添加一个"形参数量"字段来存储函数拥有的形参数量。然后，我们可以立即用这个符号后面的符号表条目来跟随每个函数形参。

当我们解析函数的形参列表时，我们可以在全局符号部分添加形参以记录函数的原型。同时，我们也可以将形参添加到局部符号部分的条目中，因为它们将被函数本身用作局部变量。

当我们需要确定函数调用的实参列表是否与函数原型相匹配时，我们可以找到函数的全局符号表条目，然后将符号表中的后续条目与实参列表进行比较。

最后，当对全局符号进行搜索时，我们可以通过加载函数的"形参数量"字段并跳过这么多符号表条目来轻松跳过函数的形参条目。

### 将形参保存在寄存器中：不可能

实际上，我在尝试实现上述内容之后才写这一部分，所以我回来重新审视了一下设计。我原以为我们能够将作为寄存器传递的形参保留在它们的寄存器中：这会使访问它们更快，并使栈帧更小。但这并不总是可能的，原因如下。考虑这段代码：

```c
void myfunction(int a) {        // a 是寄存器中的一个形参
  int b;                        // b 是栈上的一个局部变量

  // 调用一个函数来更新 a 和 b
  b= function2(&a);
}
```

如果形参 `a` 在寄存器中，我们就无法用 `&` 运算符获取它的地址。因此，我们必须将它复制到内存中的某个地方。而且，由于形参是函数的局部变量，我们需要将其复制到栈上。

有段时间我曾考虑过遍历 AST 来查看树中的哪些形参需要真实的地址，但后来我想起了我遵循的是 KISS 原则：保持简单、笨蛋！所以我会将所有形参从寄存器复制到栈上。

### 局部变量的位置

一旦形参或局部变量被复制或放置到栈上，我们如何确定它们在栈上的位置？为此，我将在每个局部符号表条目中添加一个 `posn` 字段。这将表示该变量在帧基指针下方的偏移量。

查看 C 的 BNF 语法，函数声明列表（即函数形参列表）位于局部变量的声明列表之前，而局部变量的声明列表位于语句列表之前。

这意味着，当我们解析形参然后是局部变量时，我们可以在解析语句列表之前确定它们在栈上的位置。
